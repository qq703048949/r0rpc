# R0RPC 项目总览

## 1. 项目是什么

R0RPC 是一个面向设备侧的 RPC 中继平台，整体思路类似 Sekiro RPC。

它的核心目标是：

1. 让后端服务可以按 `group` 或指定 `clientId` 把请求发到在线设备
2. 让设备执行动作后，把结果实时回传给后端
3. 提供管理后台，用来查看账号、设备、调用记录、统计指标和在线状态
4. 支持 Linux 一键部署，默认可连本机 Docker MySQL/Redis，也支持接入外部 MySQL/Redis

当前版本以 Linux 部署为主，不再围绕 Windows 本地启动维护。

## 2. 项目结构

当前仓库按职责划分：

- `cmd/`
  Go 程序入口。
  `cmd/server` 是主服务，`cmd/dbinit` 负责数据库初始化。

- `internal/`
  Go 服务核心代码。
  包含配置、鉴权、RPC 调度、存储、Web API、内置前端等。

- `deploy/linux/`
  Linux 一键部署入口。
  这里是你真正要关注的部署目录。

- `clients/java/`
  Java Relay Client SDK。
  如果你要做 Android、Xposed 或 Java 侧接入，这里是基础客户端库。

- `clients/xposed-demo/`
  Android/Xposed 示例项目。
  用来演示如何在设备端注册动作并接收后端调用。

- `examples/python/`
  Python 示例。
  包含调用示例和 WebSocket 设备端示例。

- `docs/`
  重点文档。

- `docs/archive/`
  历史会话记录、原始问答、调试过程归档。
  这些不是主说明，只是保留备查。

## 3. 系统怎么工作

系统可以理解成两端加一个中间层：

1. 调用方
   一般是你的后端服务、Python 脚本，或者其他业务系统。
   它通过 HTTP 调用 R0RPC 的接口，发起 RPC 请求。

2. R0RPC 服务端
   负责认证、查找在线设备、路由请求、等待设备返回结果、记录日志和指标。

3. 设备客户端
   设备端先登录，再通过 WebSocket 长连接保持在线。
   收到任务后执行动作，再把结果回传给服务端。

简单流程如下：

1. 设备客户端先连接到 R0RPC
2. 调用方发起 `POST /rpc/invoke/{group}/{action}`
3. 服务端根据 `group` 或 `clientId` 找到在线设备
4. 服务端把任务通过 WebSocket 推送给设备
5. 设备执行动作并返回结果
6. 服务端把结果同步返回给调用方

## 4. 当前使用的传输方式

当前主通道是 WebSocket。

设备侧流程是：

1. `POST /api/client/login`
2. 读取返回里的 `token` 和 `wsUrl`
3. 连接 `GET /api/client/ws?token=...`
4. 周期性发送 heartbeat
5. 接收任务并回传结果

这样做的优点是：

- 设备在线时延更低
- 空闲时 HTTP 开销更小
- 更适合大量设备常驻连接

## 5. 主要能力

### 5.1 管理后台

内置了后台页面，可以查看：

- 账号列表
- 分组列表
- 设备列表
- 调用记录
- 统计指标
- 在线状态

默认访问地址：

- 管理后台：`http://YOUR_SERVER_IP:9876/`
- 健康检查：`http://YOUR_SERVER_IP:9876/healthz`

### 5.2 设备按 group 调用

调用方可以按 `group` 发请求。

如果没有指定 `clientId`，服务端会在同组在线设备里做轮询分发。

如果指定了 `clientId`，就只发给指定设备。

### 5.3 按 group 查询有哪些在线设备

你提到的“根据 group 查询有哪些设备”的接口已经有，接口如下：

`GET /rpc/clientQueue?group={groupName}`

可选参数：

- `group`：必填，分组名
- `status`：可选，设备状态过滤
- `limit`：可选，返回数量上限

它会返回：

- `group`
- `count`
- `clientIds`
- `items`

其中每个 `item` 里有：

- `clientId`
- `group`
- `platform`
- `status`
- `lastSeenAt`
- `lastIp`

示例：

```bash
curl "http://127.0.0.1:9876/rpc/clientQueue?group=idlefish"
```

返回示意：

```json
{
  "group": "idlefish",
  "count": 2,
  "clientIds": ["device-a", "device-b"],
  "items": [
    {
      "clientId": "device-a",
      "group": "idlefish",
      "platform": "android",
      "status": "online",
      "lastSeenAt": "2026-04-01T16:00:00+08:00",
      "lastIp": "10.0.0.12"
    }
  ]
}
```

如果你只想看在线设备，不传 `status` 也可以，因为这个接口默认就优先返回在线设备队列。

## 6. 常用接口说明

### 6.1 管理员登录

接口：`POST /api/auth/login`

请求：

```json
{
  "username": "admin",
  "password": "123456"
}
```

返回：

- `token`
- `user`

### 6.2 发起 RPC 调用

接口：`POST /rpc/invoke/{group}/{action}`

当前支持两种认证方式：

1. 传统 Bearer Token
2. 请求体里直接带 `username/password`

这意味着你现在可以不先登录拿 token，而是直接调用。

最推荐的简单调用体：

```json
{
  "username": "admin",
  "password": "123456",
  "timeoutSeconds": 20,
  "payload": {
    "encode_str": "xxx"
  }
}
```

如果你要指定设备，可以多带一个字段：

```json
{
  "username": "admin",
  "password": "123456",
  "clientId": "device-001",
  "timeoutSeconds": 20,
  "payload": {
    "encode_str": "xxx"
  }
}
```

这些字段的含义建议直接记住：

- `username`：管理员账号
- `password`：管理员密码
- `clientId`：可选，指定某一台设备；不传就按 group 轮询在线设备
- `timeoutSeconds`：可选，请求超时时间，单位是秒；比如 `20` 就表示最多等 20 秒。`不传` 时走服务端默认值，当前默认是 `25` 秒
- `payload`：真正发给设备 action 的参数对象，建议直接写扁平结构，不要再多套一层 `payload`

调用示例：

```bash
curl -X POST "http://127.0.0.1:9876/rpc/invoke/idlefish/decrypt" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "admin",
    "password": "123456",
    "timeoutSeconds": 20,
    "payload": {
      "encode_str": "v7eNwdELBmc1hOkagpP6NQ=="
    }
  }'
```

### 6.3 调用返回结构

服务端现在返回统一结构，重点看这些字段：

- `is_ok`
- `status`
- `httpCode`
- `requestId`
- `clientId`
- `data`
- `error`

判断规则建议直接看 `is_ok`：

- `true`：本次调用成功
- `false`：本次调用失败

成功示例：

```json
{
  "is_ok": true,
  "requestId": "xxx",
  "group": "idlefish",
  "action": "decrypt",
  "clientId": "a9178e59d1d77782",
  "requestPayload": {
    "encode_str": "v7eNwdELBmc1hOkagpP6NQ=="
  },
  "status": "success",
  "httpCode": 200,
  "data": {
    "data": "解密结果"
  },
  "latencyMs": 132,
  "error": ""
}
```

超时示例：

```json
{
  "is_ok": false,
  "status": "timeout",
  "httpCode": 504,
  "requestId": "xxx",
  "group": "idlefish",
  "action": "decrypt",
  "clientId": "a9178e59d1d77782",
  "requestPayload": {
    "encode_str": "v7eNwdELBmc1hOkagpP6NQ=="
  },
  "error": "context deadline exceeded"
}
```

## 7. 设备端接入方式

### 7.1 Java / Xposed

看这里：

- [clients/java/README.md](r0rpc/clients/java/README.md)
- [clients/xposed-demo/README.md](r0rpc/clients/xposed-demo/README.md)

当前 Java/Xposed 的 handler 写法是：

1. 用 `registerHandler("action", handler)` 注册动作
2. handler 内只实现 `handleRequest(...)`
3. 请求参数推荐走扁平 `payload`
4. 需要时可用 `@AutoBind` 直接把 payload 字段绑定到成员变量

示意：

```java
relayClient.registerHandler("decrypt", new DecryptHandler());
```

```java
public class DecryptHandler implements RelayHandler {
    @AutoBind
    private String encode_str;

    @Override
    public void handleRequest(RelayRequest request, RelayResponse response) throws Exception {
        response.success(encode_str);
    }
}
```

## 8. Linux 部署方式

这是当前项目最重要的部分。

### 8.1 支持的 Linux 系统

自动安装脚本目前支持：

- Ubuntu
- Debian
- CentOS
- RHEL
- Rocky Linux
- AlmaLinux
- Oracle Linux

### 8.2 一键部署入口

默认最快方式：

```bash
cd deploy/linux
chmod +x quickstart.sh
./quickstart.sh
```

这个脚本会做：

1. 检查 `docker` 和 `docker compose`
2. 如果缺失则自动安装
3. 如果没有 `.env.docker` 就从 `.env.example` 自动生成
4. 按当前模式启动 MySQL / Redis / R0RPC
5. 等待依赖服务就绪
6. 自动初始化数据库
7. 启动服务端

### 8.3 默认部署模式

默认配置是：

```text
MYSQL_MODE=builtin
REDIS_MODE=builtin
```

这表示：

- MySQL 用本机 Docker 启动
- Redis 用本机 Docker 启动
- R0RPC 也在 Docker 中启动

也就是默认整套服务都在 Linux 机器本地拉起，最省事。

### 8.4 使用你自己的 MySQL / Redis

如果你已经有自己的 MySQL / Redis，只需要改 `deploy/linux/.env.docker`：

```text
MYSQL_MODE=external
REDIS_MODE=external
MYSQL_HOST=10.0.0.12
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=your-password
MYSQL_DB=r0rpc
REDIS_ADDR=10.0.0.13:6379
REDIS_PASSWORD=your-password
REDIS_DB=0
```

然后执行：

```bash
cd deploy/linux
./deploy.sh
```

### 8.5 混合模式

也可以一部分内置，一部分外部。

例如：

```text
MYSQL_MODE=builtin
REDIS_MODE=external
REDIS_ADDR=10.0.0.13:6379
REDIS_PASSWORD=your-password
REDIS_DB=0
```

### 8.6 数据库初始化是否需要手动处理

不需要。

当前 Linux 启动链路已经自动处理：

1. 检查 MySQL 是否可连
2. 检查 Redis 是否可连
3. 自动执行 `/app/r0rpc-dbinit`
4. 自动创建数据库和表
5. 自动确保管理员账号存在
6. 然后再启动 `/app/r0rpc-server`

所以你不用手工建库、建表、导入 SQL。

### 8.7 常用命令

安装 Docker 和 Compose：

```bash
cd deploy/linux
./install-docker.sh
```

部署：

```bash
cd deploy/linux
./deploy.sh
```

停止：

```bash
cd deploy/linux
./stop.sh
```

## 9. 默认账号与访问方式

默认管理账号：

- 用户名：`admin`
- 密码：`123456`

如果你在 `deploy/linux/.env.docker` 里改了：

- `BOOTSTRAP_ADMIN_USERNAME`
- `BOOTSTRAP_ADMIN_PASSWORD`

那就以你自己的配置为准。

## 10. 适合什么场景

这个项目比较适合：

- 设备在线执行远程动作
- 按 group 管理设备池
- 后端需要把任务实时推送到设备
- 需要管理后台查看在线情况、调用记录和趋势指标
- Android / Xposed / Java 设备接入
- Python 脚本快速联调调用

## 11. 建议的使用顺序

如果你是第一次接触这个项目，建议这样用：

1. 先看 [deploy/linux/README.md](r0rpc/deploy/linux/README.md)
2. 在 Linux 上跑 `deploy/linux/quickstart.sh`
3. 打开后台确认服务正常
4. 用 `GET /rpc/clientQueue?group=...` 看某个分组下有哪些在线设备
5. 用 `POST /rpc/invoke/{group}/{action}` 发一次测试请求
6. 再根据你的接入方式选择 Java/Xposed 或 Python 示例
